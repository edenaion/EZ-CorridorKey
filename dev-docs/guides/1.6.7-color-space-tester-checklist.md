# 1.6.7 Tester Checklist

Use this checklist for the `release/1.6.7-user-test` branch.

Temporary build verification for this tester branch:

- Launch the app and confirm the title bar or `Help > About` shows `v1.6.7.3 test`.
- If available, also confirm the short git hash shown in the app matches the latest tester-branch commit.
- Remove this visible tester build identifier before merging the branch back into `main`.

## Goals

- Confirm the left `INPUT` viewer matches CorridorKey's current source interpretation.
- Confirm input thumbnails match the left viewer.
- Confirm manual color-space overrides stay sticky when reselecting clips in the tray.
- Confirm `Import Alpha` does not reset color space or replace the chosen right-view mode with mismatched pixels.
- Confirm live preview behaves correctly on a fresh launch.
- Confirm export thumbnails match the saved output they represent.
- Confirm exports only change after rerunning inference.
- Confirm `Processed EXR` round-trips back into Resolve without unintended brightness, gamma, or contrast shifts.

## Test 1: sRGB source

1. Launch the app fresh.
2. Load a normal video or PNG/JPG sequence that should be treated as `sRGB`.
3. Select `sRGB` in the `Color Space` dropdown.
4. Confirm the left `INPUT` viewer looks correct.
5. Confirm the input thumbnail matches the left viewer.
6. Toggle to `Linear`.
7. Confirm the left viewer changes immediately.
8. Confirm the input thumbnail updates to match.
9. Toggle back to `sRGB`.

Expected:
- Left viewer and input thumbnail always match each other.
- The right viewer updates if `Live Preview` is on and the clip is `READY` or `COMPLETE`.

## Test 2: linear source

1. Load a true linear EXR sequence.
2. Select `Linear`.
3. Confirm the left `INPUT` viewer looks correct.
4. Confirm the input thumbnail matches.
5. Switch to `sRGB`.
6. Confirm the left viewer and thumbnail both change in the same way.

Expected:
- No mismatch between left viewer and input thumbnail.
- The chosen interpretation is visually obvious and consistent.

## Test 3: clip selection keeps the chosen color space

1. Load a clip.
2. Change `Color Space` to the non-default choice for that clip.
3. Click the clip thumbnail in the lower-left tray again, or switch to another clip and back.
4. Re-open the original clip.

Expected:
- The `Color Space` dropdown stays on the value you chose.
- The left `INPUT` viewer does not snap back to the auto-detected interpretation.
- The input thumbnail still matches the left viewer after reselection.

## Test 4: import alpha does not scramble preview state

1. Load a clip and choose the intended `Color Space`.
2. Set the right viewer to `INPUT` or `ALPHA`.
3. Use `IMPORT ALPHA` and bring in a valid matte sequence.
4. If live preview starts running, watch the right viewer while the import completes.

Expected:
- The `Color Space` dropdown does not reset itself.
- `INPUT` mode stays true input pixels and is not replaced by a dark checkerboard comp.
- `ALPHA` mode stays an alpha preview.
- The mode label and the pixels shown agree with each other.

## Test 5: fresh-launch live preview

1. Fresh-launch the app.
2. Open a `READY` or `COMPLETE` clip with `Live Preview` enabled.
3. Change `Color Space`, `Despill`, or `Refiner`.

Expected:
- The first update may pause briefly while the engine loads.
- It should not silently fail or require a prior inference run just to preview.

## Test 6: export behavior and export thumbnails

1. Run inference with `Color Space = sRGB`.
2. Inspect the right viewer and the export thumbnail.
3. Change `Color Space` to `Linear` without rerunning inference.
4. Inspect the saved export thumbnail and files again.
5. Rerun inference.
6. Switch right-view modes such as `COMP` and `PROCESSED` and confirm the export tray updates to the saved output, not the input card.

Expected:
- Before rerun: viewer/live preview may change, but saved files on disk do not.
- After rerun: new exports reflect the newly selected color space.
- The export thumbnail matches the saved output mode it represents.
- The export thumbnail should not simply mirror the input thumbnail unless the saved output genuinely looks the same.

## Test 7: Processed EXR round-trip in Resolve

This test is especially important on Apple Silicon / MLX.

1. Use a real clip the tester can also inspect in DaVinci Resolve.
2. Pick the intended `Color Space`.
3. Run inference and export `Processed` as `EXR`.
4. Import the saved `Processed EXR` back into Resolve with no extra grade or color adjustment.
5. Compare the keyed result against the original source using a wipe or side-by-side inspection.
6. If needed, repeat once with `Despill = 0.0` to isolate export math from creative/keying changes.

Expected:
- No unexpected darkening, gamma shift, or contrast shift from the export itself.
- Any visible change should be explainable by despill, refiner, or matte cleanup, not by the file being exported and re-imported.
- The result should not look globally darker just because it came through CorridorKey.

## Test 8: parallel frames UI

1. Open the main inference panel.
2. Hover `Parallel frames`.
3. Open `Preferences`.

Expected:
- The tooltip says CUDA-only and not supported on Apple Silicon right now.
- There is no separate `Performance` section in Preferences.
